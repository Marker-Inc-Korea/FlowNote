# FlowNote 제품 개요

## 1. 목적

FlowNote는 생산공장 현장에서 사용하는 기술문서, 도면, 매뉴얼, 작업지시서 등 다양한 업무 문서와 현장 지식을 안정적으로 관리하는 독립형 서버이다.

문서 파일과 문서 메타데이터를 분리해서 저장하고, 동일 문서의 수정은 새 버전으로 누적한다. 업로드된 문서를 항상 최신 확정본으로 보지 않고 작업중, 검토중, 현장 공개, 보관 상태를 구분한다.

FlowNote는 문서 관리에 더해 현장 코멘트, 작업 평가, 문제점 등록, 보고서 생성 보조, AI 검색과 작업 조언을 제공한다. 초기에는 완벽한 DMS나 KMS를 목표로 하지 않는다. 문서 증거와 현장 경험, 작업 맥락을 오래 운영하면서 연결해 생산현장의 노하우와 생산보고서를 축적하고, 향후 AI가 의사결정 조언에 활용할 수 있는 현장 데이터 기반을 쌓는 것을 1차 목표로 한다.

FlowNote는 MES나 ERP를 대체하지 않는다. 초기 작업지시 구조는 MES 연동이 아니라 관리자가 직접 입력하고 고객이 이미 사용하는 문서 정리 구조에 연결하는 방식으로 운영한다. 이미 MES를 구축한 공장, ERP를 MES처럼 사용하는 공장, 아직 별도 시스템이 없는 공장을 모두 고려하되, 기존 MES/ERP 연동은 후속 단계에서 작업지시, 품목, 공정, 설비, 생산실적 같은 정형 데이터를 자동 수신하는 방식으로 확장한다.

FlowNote는 사내 서버형 운영을 전제로 한다. 생산현장의 기술문서, 도면, 생산규격, 작업 보고서는 외부 클라우드 사용을 꺼리는 경우가 많으므로, 초기 배포는 고객 현장 또는 사내 서버 PC 1대에 서버를 설치하고 현장 PC에는 클라이언트 설치파일을 배포하는 방식을 기준으로 한다.

문서 구조는 프로그램이 제시하거나 강제하지 않는다. 고객마다 이미 문서를 정리하는 기준과 현장 용어가 있으므로, FlowNote는 그 구조를 흡수해야 한다. 현장에서 말하는 BOM 문서 구조는 MES의 BOM을 차용한다는 뜻이 아니라, 문서를 계층적으로 묶어 부르는 현장 용어의 예시이다. 이 표현은 어디까지나 예시이며, FlowNote의 기본 구조나 필수 구조가 아니다.

현장 사용자가 처음 체감해야 하는 가치는 복잡한 지식관리나 AI가 아니라, 익숙한 탐색기형 목록에서 필요한 문서와 사진 기록을 찾고, 현장 공개 버전을 바로 열람하며, 현재 작업순서를 큰 화면에서 확인하는 것이다. FlowNote는 문서관리 기능을 기반으로 하되, 현장 뷰어, 검색, 사진 기록, 작업순서 공유를 초기 체감 기능으로 둔다.

현장의 소리는 FlowNote에서 중요한 관리 대상이다. 사용자의 짧은 경험, 불편, 문제점, 개선 의견을 현재 문서와 작업 맥락에 연결해 기록함으로써 특정 작업자에게만 남아 있던 지식이 조직의 지식으로 발전하도록 한다.

다만 현장 코멘트는 원천 이력이지 곧바로 확정 데이터가 아니다. 생산 중 쌓이는 코멘트와 문제점은 관리자급 사용자의 검토, 분석, 정리 과정을 거쳐 보고서 형태의 정제 문서와 연결되어야 한다. 원천 이력만 있으면 문서화된 판단 근거가 약하고, 보고서만 있으면 어떤 현장 맥락에서 나온 결론인지 추적하기 어렵다. FlowNote는 두 데이터를 함께 유지한다.

## 2. 한 문장 설명

FlowNote는 생산현장의 HWP, Word, PowerPoint, Excel, PDF, DWG 등 다양한 문서를 버전 관리하고, 현장 단말기 열람, 관리자 파일 변경 감지, 현장 지식 축적, AI 검색과 작업 조언을 지원하는 독립형 문서·현장지식 관리 서버이다.

## 3. 사용자와 역할

| 역할 | 주요 목적 | 허용 기능 |
| --- | --- | --- |
| 현장 사용자 | 현장 공개 문서 열람과 현장 코멘트 등록 | view, comment |
| 관리자 | 문서 등록, 개정, 삭제, 권한 관리 | read, download, write, delete, manage |
| 네이티브 클라이언트 | WPF 또는 Avalonia 기반 현장/관리자 클라이언트 | viewer, comment, admin |
| Python FastAPI 서버 | 앱과 외부 시스템이 사용하는 백엔드 | auth, document, field note, integration |
| 외부 시스템 | MES/ERP/기타 업무 시스템 연동 | API 연동 |

현장 사용자 단말기는 뷰어 전용 모드로 동작하며 파일 감시와 문서 업로드를 수행하지 않는다.

클라이언트 UI는 WPF 또는 Avalonia 기반 설치형 네이티브 앱을 기본으로 한다. 고객 생산공장 운영 환경에서는 일반 브라우저를 통한 직접 웹 접속을 기본 사용 방식으로 두지 않는다. 독립 Web UI는 신규 개발 대상에서 제외하며, 화면은 앱에서 구현하고 Python FastAPI 서버와 REST API로 통신한다.

문서 미리보기나 외부 뷰어 연계가 필요한 경우 앱 내부의 제한된 표시 컴포넌트를 사용할 수 있지만, 이는 공통 웹 프론트엔드 개발을 뜻하지 않는다. 로컬 제어와 파일 감시처럼 운영체제 기능이 필요한 부분은 각 앱의 네이티브 기능으로 처리한다.

메타데이터 DB는 SQLite를 우선 사용한다. 현장 규모나 동시 사용자가 늘어 SQLite로 부족해지면 PostgreSQL로 확장한다. 파일은 서버 PC의 로컬 `storage/` 폴더에 저장한다. 초기 배포 단위는 서버 PC 1대와 클라이언트 설치파일이다.

## 4. 문서 범위

FlowNote에서 문서는 생산 현장에서 사용하는 모든 업무 파일을 뜻한다.

- HWP
- Word: DOC, DOCX
- PowerPoint: PPT, PPTX
- Excel: XLS, XLSX
- PDF
- CAD 도면: DWG
- 기타 기술문서, 도면, 매뉴얼, 작업지시서 파일

초기 설계의 우선순위는 모든 형식의 저장, 열람, 버전 관리, 문서 상태, 태그 기반 연결이다. 관리자 다운로드와 형식별 미리보기, 변환은 후속 클라이언트 앱 또는 뷰어 계층에서 확장한다.

## 5. 주요 기능

### 5.0 제품 균형 기준

- 문서관리 기능은 최신본, 버전, 권한, 이력, 열람 보안을 책임진다.
- 현장지식 기능은 코멘트, 문제점, 작업 평가, 관리자 정리 문구, 보고서 원천 데이터를 책임진다.
- 작업 맥락 기능은 고객이 정한 문서 정리 구조, 작업지시서, 작업내역 버전을 문서와 연결한다.
- 현장 체감 기능은 탐색기형 검색, 보기 전용 뷰어, 사진 기록, 작업순서판, 현장 TV 화면을 우선한다.
- AI 기능은 위 데이터를 검색하고 조언하기 위한 활용 계층이며, 초기 단계에서는 검색과 작업 조언을 중요 기능으로 둔다.
- 과거 생산 데이터 기반의 신규 작업 설계나 자동 의사결정은 장기 가능성으로만 두고, 현재 범위에서는 핵심 기능으로 내세우지 않는다.
- 현장 사용자의 경험은 관리 대상 데이터이며, 현장의 문제점과 개선 의견을 기록으로 남기는 것을 제품의 중요한 가치로 둔다.
- 현장 코멘트는 관리자 분석을 거쳐 정제 보고서와 연결되어야 실제 활용 가능한 데이터가 된다.
- AI는 보고서 작성을 대체하지 않고, 관리자급 사용자가 원천 코멘트와 작업 이력을 정리해 보고서 초안을 작성하는 것을 보조한다.
- MES/ERP가 가진 정형 데이터와 FlowNote가 수집하는 현장 노하우/문서/보고서를 함께 활용한다.
- FlowNote는 기존 MES/ERP의 대체재가 아니라 AI 활용 데이터를 보강하는 보완 계층이다.
- 배포 방식은 고객 데이터 주권을 우선하며, 설치 위치와 접근 경로는 고객 보안 정책에 따라 결정한다.
- 설비, 품목, 공정, 오류 유형 같은 태그를 문서와 현장 코멘트에 붙여 고객 정의 문서 구조에 흡수되지 못한 관계를 보완한다.
- 위키, 게시판, 자유 지식베이스처럼 문서와 작업 맥락에서 분리된 지식관리 기능은 초기 범위로 두지 않는다.

### 5.1 문서 관리

- 문서 등록
- 파일 업로드
- 문서 목록과 상세 조회
- 현장 공개 버전 열람
- 관리자 다운로드 정책. 클라이언트 앱 단계
- 특정 버전 조회
- 문서 메타데이터, 태그, 연결 대상 관리
- 설비, 품목, 공정, 오류 유형 등 태그 기반 관계 보완
- 고객 정의 문서 구조 구성
- 트리형, 목록형, 프로젝트형, 작업지시서형 등 현장별 정리 방식 지원
- 문서 상태와 보존 정책 관리
- 작업중, 검토중, 공개, 보관 등 문서 상태 구분
- 문서 변경 이력과 접근 로그 기록
- 파일명, 문서명, 태그, 문서 구조, 작업지시 기준의 탐색기형 기본 검색
- PDF와 이미지 중심의 우선 뷰어 제공
- Excel, PowerPoint, CAD 원본은 저장과 이력 관리를 우선하고, 현장 열람은 PDF 변환본 또는 지원 가능한 미리보기 계층을 우선 사용

### 5.2 버전 관리

- 동일 문서를 `Document`로 유지
- 수정 파일은 `DocumentVersion`으로 추가
- 새 버전 등록 시 변경 사유 필수
- 최근 등록 버전과 현장 공개 버전을 분리해서 관리
- 버전 상승 시 현장 단말기 알림 생성

### 5.3 현장 단말기

- 클라이언트는 WPF 또는 Avalonia 기반 네이티브 앱으로 작성한다.
- 클라이언트 앱은 현장 뷰어와 현장 입력 UI를 앱 화면으로 제공한다.
- 현장 공개 문서 열람
- 문서 변경 알림 확인
- 현재 문서에 연결된 현장 코멘트 등록
- 신호등 방식의 단순 상태 기록
- 정형 문구 선택 방식의 빠른 입력
- 필요 시 짧은 직접 입력
- AI가 제공한 작업 전 주의 사항 확인
- 설정된 제한 시간이 지나면 문서 뷰어 자동 닫힘. 클라이언트 앱 단계

현장 사용자 단말기는 파일을 감시하지 않는다. 현재 단계에서 문서 열람 자체를 과도하게 막지는 않는다. 직접 다운로드 차단과 운영 설정 기반 뷰어 자동 닫힘은 Android/Windows 클라이언트 앱 단계에서 구현한다.

### 5.4 관리자 파일 변경 감지

- 관리자 화면도 같은 네이티브 클라이언트 안에서 권한에 따라 제공한다.
- 관리자급 사용자가 지정한 로컬 파일 또는 폴더 감시
- 변경 시간, 크기, 해시값 기반 변경 감지
- 변경된 파일을 새 버전 업로드 후보로 표시
- 관리자가 변경 사유와 버전명을 입력한 뒤 업로드 확정

파일 감시는 자동 업로드가 아니라 업로드 보조 기능이다.

파일 감시는 서버가 직접 수행하지 않는다. 클라이언트 앱이 로컬 파일 감시, 파일 선택, 업로드 보조를 수행하고 Python FastAPI 서버에 변경 후보와 업로드 결과를 등록한다.

### 5.5 현장 코멘트와 작업 평가

- 현재 문서 또는 문서 버전에 연결된 코멘트 등록
- 설비, 품목, 공정, 오류 유형 등 태그 연결
- 신호등 방식의 단순 기록: 정상, 주의, 문제 등
- 직접 입력 방식 지원
- 정형 문구 선택 방식 지원
- 현장 사용자가 말로 전달한 내용을 관리자가 대신 입력하는 방식 지원
- 현장 원문과 관리자 정리 문구 분리
- 문제점, 작업 평가, 개선 의견 분류
- 관리자 분석 결과와 연결
- 보고서 생성 원천 데이터로 활용
- 코멘트 등록자와 실제 전달자 또는 작업자 추적
- 사진 첨부 기반 현장 기록과 일일 작업일지 원천 데이터 관리

현장 사용자가 실시간으로 많은 내용을 입력하는 것은 작업 흐름에 직접 들어가야 하므로 초기 안착이 어렵다. 따라서 초기에는 신호등식 기록, 정형 문구 선택, 짧은 메모, 관리자 대리 입력을 허용하고, MES 연동이나 자동화 입력은 단계적으로 확장한다.

현장 코멘트는 난잡한 데이터가 될 수 있음을 전제로 한다. 관리자는 여러 코멘트, 작업내역, 관련 문서, 태그를 묶어 분석하고, AI는 이 분석을 보고서 초안으로 정리하는 데 도움을 준다. 최종 보고서는 다시 FlowNote 문서로 저장하고 원천 코멘트와 연결해 추적성을 유지한다.

### 5.5.1 작업자와 추적성

- 문서 수정자, 버전 등록자, 문서 열람자를 기록한다.
- 현장 코멘트의 입력자와 실제 전달자 또는 작업자를 구분한다.
- 실제 작업자 개인을 항상 식별하지 못하는 현장에서는 작업반, 라인, 조장, 관리자 대리 등록자를 기록할 수 있게 한다.
- 작업자 정보는 개인 추궁보다 문서 효율성, 교육, 재발 방지, 이력 추적을 위한 근거로 사용한다.

### 5.6 작업내역과 AI

- 작업지시 문서와 관련 문서 연결
- 고객 정의 문서 구조와 관련 문서 연결
- 라인, 공정, 설비 같은 생산 맥락 연결
- 작업내역 버전 관리
- 등록 문서와 작업내역 기반 AI 검색
- 작업지시 문서 기반 위험 요소 미리보기
- 과거 문제점과 현장 코멘트 기반 작업 조언
- AI 답변의 근거 문서와 버전 표시

### 5.7 MES/ERP 연동

- 기존 MES 또는 ERP의 작업지시, 품목, 공정, 설비, 생산실적 데이터를 API로 수집
- 현장별 시스템 차이를 고려한 연동 어댑터 구성
- 외부 시스템의 원본 데이터를 FlowNote가 임의로 대체하지 않음
- 외부 작업지시와 FlowNote 문서 구조, 현장 코멘트, 작업내역 연결
- 초기에는 작업지시와 문서 구조를 관리자가 직접 입력
- MES/ERP 연동이 추가되면 관리자 입력 구조를 외부 데이터 자동 수신 구조로 확장

### 5.8 작업순서판과 현장 TV 화면

- 사무실 또는 관리자급 사용자가 작업지시와 작업내역의 현장 처리 순서를 조정
- 작업순서는 라인, 공정, 설비, 품목, 작업반 같은 현장 기준으로 필터링
- 현장 사용자는 승인된 단말기 또는 대형 TV 화면에서 현재 순서를 확인
- 반장, 조장, 관리자 권한으로 순서 변경, 보류, 완료 처리 가능
- 순서 변경, 상태 변경, 확인 이력은 감사 로그와 알림으로 관리
- MES가 순서를 충분히 판단하지 못하는 현장에서는 FlowNote가 사무실 판단과 현장 실행 사이의 공유 화면 역할을 수행

## 6. 설계 원칙

- 문서 파일과 메타데이터 DB를 분리한다.
- 문서, 파일, 버전을 분리한다.
- 문서 ID는 외부 시스템에서 안정적으로 참조 가능해야 한다.
- 업로드된 문서가 항상 최신 확정본이라고 가정하지 않는다.
- 현장 공개 대상 문서와 작업중 문서를 상태로 구분한다.
- 문서는 고객이 정한 업무 구조에 연결될 수 있어야 한다.
- 프로그램은 문서 구조를 강제하지 않고, 고객의 기존 정리 방식을 흡수해야 한다.
- 트리 구조와 현장 용어의 BOM 문서 구조는 지원 가능한 예시이며 기본 강제 구조가 아니다.
- 현장 코멘트와 작업 문제점은 문서, 문서 버전, 고객 정의 문서 구조의 항목, 작업내역과 연결되어야 한다.
- 원천 코멘트와 관리자 분석 보고서는 서로 연결되어야 한다.
- 고객 정의 구조로 표현되지 않는 관계는 태그로 보완한다.
- 문서관리와 현장지식관리 중 어느 한쪽으로 제품 방향이 치우치지 않도록 한다.
- MES/ERP는 대체 대상이 아니라 연동 대상이다.
- 작업지시는 초기에는 관리자 입력 기준이며, MES/ERP 연동은 후속 자동화 확장으로 둔다.
- 라인, 공정, 설비, 품목 개념은 생산현장 맥락을 표현하기 위해 차용하되, 고객별 용어와 구조에 맞게 유연하게 매핑한다.
- 외부 시스템의 정형 데이터와 FlowNote의 현장지식 데이터를 연결해 AI 활용성을 높인다.
- 초기 AI 목표는 자동 판단이 아니라, 추후 AI 검색과 조언에 사용할 수 있는 정합성 있는 데이터를 축적하는 것이다.
- 물리 삭제보다 상태 변경과 보존 정책을 우선한다.
- 회원관리 시스템을 기준으로 로그인, 인증, 역할 기반 권한을 적용한다.
- 현장 사용자는 열람과 코멘트 중심으로 제한한다.
- 관리자급이 아닌 사용자 다운로드 차단은 클라이언트 앱 단계에서 구현한다.
- 문서 뷰어 자동 닫힘은 클라이언트 앱 단계에서 구현한다.
- 관리자 기능은 권한 검사를 전제로 한다.
- AI 검색과 조언은 사용자가 접근 가능한 데이터만 참조한다.
- AI는 최종 판단을 대체하지 않고 작업 전 확인을 보조한다.
- AI는 장기적으로 의사결정 조언으로 확장할 수 있으나, 현재 구축 목표는 검색, 근거 제시, 작업 전 주의 사항 제공이다.
