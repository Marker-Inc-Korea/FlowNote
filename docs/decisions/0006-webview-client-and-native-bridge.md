# 0006. WebView 중심 클라이언트와 네이티브 브릿지

## 상태

Accepted

## 결정

FlowNote 클라이언트 UI는 웹 기술을 기본으로 한다.

플랫폼 기준은 다음과 같다.

- Android: Kotlin + WebView
- Windows: WPF + WebView2
- Web: TypeScript + React + Vite 기반 공통 웹 UI

Android와 Windows 앱은 웹 UI를 WebView로 표시한다. 고객 생산공장 운영 환경에서는 일반 브라우저 직접 접근을 기본 사용 방식으로 두지 않고, 승인된 클라이언트 앱의 WebView를 통해 접근한다. 로컬 제어, 파일 감시, 파일 선택, OS 알림처럼 클라이언트 앱 성능이나 운영체제 접근이 필요한 기능은 네이티브 앱에서 처리하고, 웹 UI와 네이티브 앱은 브릿지로 통신한다.

## 근거

네이티브 앱은 성능 면에서 장점이 있지만, UI/UX 요구사항 변화에 빠르게 대응하기 어렵다. 생산공장은 보안을 중요하게 다루므로 사용자 단말기에 문서 파일이 동기화되는 구조도 최소화해야 한다.

TypeScript + React + Vite 기반 웹 UI를 기본으로 하면 문서 뷰어, 관리자 콘솔, 현장 입력 화면을 서버 중심으로 빠르게 개선할 수 있다. 이 선택은 공장 사용자가 브라우저로 직접 접속하게 하기 위한 결정이 아니라 UI 유지보수성을 높이기 위한 결정이다. 동시에 WebView 앱을 사용하면 로컬 파일 감시나 OS 연동처럼 웹만으로 처리하기 어려운 기능을 네이티브에서 제공할 수 있다.

## 브릿지 원칙

- 브릿지는 웹 UI가 직접 접근할 수 없는 로컬 기능만 제공한다.
- 운영 환경에서는 승인된 클라이언트 앱의 WebView 접근을 기본으로 한다.
- 현장 사용자 단말기에서는 파일 감시 브릿지를 비활성화한다.
- 관리자 단말기에서만 지정 파일 또는 폴더 감시를 허용한다.
- 브릿지는 문서 파일 자동 동기화 수단으로 사용하지 않는다.
- 브릿지 호출은 권한과 단말기 모드 검사를 거친다.
- 중요한 브릿지 호출은 감사 로그 대상으로 고려한다.

## 결과

- Android 앱은 Kotlin + WebView 기준으로 설계한다.
- Windows 앱은 WPF + WebView2 기준으로 설계한다.
- TypeScript + React + Vite 웹 UI는 공통 클라이언트 화면의 기준 구현이 된다.
- 로컬 파일 감시와 파일 선택은 네이티브 브릿지 기능으로 분리한다.
- 서버 API와 WebView 브릿지 계약을 문서에서 구분한다.
