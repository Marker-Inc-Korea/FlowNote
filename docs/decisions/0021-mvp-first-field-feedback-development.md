# 0021. MVP 우선 현장 반영 개발 원칙

## 상태

Accepted

## 배경

FlowNote는 생산현장의 문서와 현장 지식을 함께 다루는 제품이다. 초기 단계에서 가장 중요한 것은 완성형 DMS/KMS를 설계하는 것이 아니라, 현장 사용자가 문서를 찾고 보고 기록하는 최소 흐름을 실제로 사용하는지 빠르게 확인하는 것이다.

현장에서는 장갑 낀 손, 터치 장비, 별도 단말기, 기존 MES/ERP 연동, 현장별 문서 정리 방식처럼 다양한 요구가 나올 수 있다. 그러나 초기부터 모든 사용 환경과 장비 투자를 전제로 설계하면 MVP 속도가 늦어지고, 실제로 필요한 기능인지 검증하기 어렵다.

## 결정

- FlowNote 초기 개발은 MVP 방식으로 기능을 빠르게 구현하고 실제 사용 여부를 검증하는 것을 우선한다.
- 기술 검증을 별도 선행 단계로 오래 두지 않고, 선택한 기술을 먼저 적용해 기능을 만들면서 한계와 보완점을 확인한다.
- 현장의 이야기는 최대한 제품 흐름에 반영한다.
- 현장 요청이 Python FastAPI, WPF/Avalonia 네이티브 클라이언트, 사내 서버형 배포, 보안 정책과 맞지 않으면 즉시 반영하지 않고 검토 후 결정한다.
- 기존 시스템과의 연동은 초기 핵심 기능을 막지 않는 범위에서 참조 ID와 기본 매핑 구조만 준비하고, 현장별 상세 연동은 후속 단계에서 맞춘다.
- 장갑 낀 손, 터치 장비, 별도 현장 단말기 같은 사용 형태는 초기부터 고정 조건으로 단정하지 않는다.
- 별도 장비 투입은 기능 검증 목적의 실험으로만 다루며, 실제 구매나 상시 운영은 공장과 비용 투입 여부를 회의한 뒤 결정한다.
- UI는 가장 직관적인 흐름을 우선하고, 현장 사용자가 실제로 기능을 쓰는지 먼저 검증한다.

## 영향

- `docs/implementation-roadmap.md`의 착수 원칙은 빠른 MVP 구현과 현장 사용 검증을 기준으로 해석한다.
- `docs/mvp-scope.md`의 성공 기준은 기능 구현 완료뿐 아니라 실제 현장 사용 여부 확인까지 포함한다.
- 초기에는 고급 장비, 특수 입력 장치, MES/ERP 상세 연동보다 문서 열람, 검색, 짧은 기록, 사진 기록, 작업순서 확인의 사용성을 우선한다.
- 현장 요청은 중요 입력으로 남기되, 제품 기술 방향과 충돌하는 경우 ADR 또는 후속 검토 항목으로 분리한다.
