# FlowNote 구현 로드맵

## 0. 현재 코드 위치

현재 코드는 Windows WPF 로컬 SQLite 프로토타입과 FastAPI SQLite MVP API로 나뉜다. WPF 앱은 로그인, 탐색기형 폴더 트리, 로컬 파일 업로드/Drag & Drop, SQLite 문서 등록, 서버 URL 설정 시 로컬 등록 직후 인증된 FastAPI 문서 등록 후보 시도, 파일 보기, FieldNote, FieldNote 첨부, 알림함, 이력, 작업순서판 편집/TV 화면까지 구현되어 있다. FastAPI 서버는 `GET /`, 헬스체크, MVP 로그인, `/auth/me`, 문서 등록/버전 등록, 문서 상태와 공개 버전 관리, 서버 로컬 `storage/` 저장, 태그, FieldNote와 첨부, 문서 접근 로그, 작업순서판 API, Bearer 인증, role 기반 권한 검사가 구현되어 있다. WPF 서버 API 클라이언트는 `FLOWNOTE_API_BASE_URL`이 설정된 스모크 테스트에서 서버 로그인 후 인증 헤더가 붙은 `/auth/me`, 서버 문서 등록/목록/버전 조회, 문서 버전에 연결된 서버 FieldNote 등록, FieldNote 첨부 등록/조회, 문서 접근 로그 등록/조회, 작업순서판 생성/항목 추가/정렬/상태 변경/이력 조회를 검증한다.

따라서 이 로드맵의 운영용 토큰 갱신/폐기, 작업순서판 알림 발송 정책, 보고서, AI, MES/ERP 항목은 앞으로 구현할 목표이다. 미래에 만들 기능은 현재 코드와 비교해 구현 누락으로 판단하지 않는다.

## 1. 기본 전략

FlowNote는 Python FastAPI 서버와 Windows WPF 클라이언트를 기준으로 구현한다.

구현 방향은 문서관리 기능만 먼저 만들고 현장지식은 나중에 붙이는 방식으로 보지 않는다. 1차부터 문서, 고객 정의 문서 구조, 현장 코멘트가 연결되는 최소 구조를 넣어 향후 AI 검색과 작업 조언의 근거 데이터가 자연스럽게 쌓이도록 한다.

현장 체감 우선순위는 AI보다 탐색기형 검색, 보기 전용 뷰어, 사진 기록, 작업순서판과 현장 TV 화면이다. 사용자는 윈도우 탐색기처럼 찾고, 바로 보고, 필요한 사진 기록을 남기고, 현재 처리 순서를 큰 화면에서 확인할 수 있어야 한다.

FlowNote는 MES/ERP를 대체하는 방향으로 개발하지 않는다. 초기 작업지시는 관리자가 직접 입력하는 구조로 시작한다. 기존 MES/ERP가 있으면 후속 단계에서 작업지시, 품목, 공정, 설비, 생산실적 같은 정형 데이터를 자동 수신하도록 확장한다.

과거 생산 기반으로 새로운 작업을 설계하거나 자동 의사결정을 지원하는 기능은 장기 가능성으로 둔다. 현재 구축의 중심은 현장의 소리, 사진 기록, 인수인계, 작업순서, 문서 열람 이력을 안정적으로 남겨 후속 AI 활용의 근거 데이터를 쌓는 것이다.

Windows WPF 클라이언트는 실제 운영 프론트엔드이다. 현장 사용자 열람과 코멘트 입력, 관리자 화면, 파일 감시 업로드 보조를 같은 설치형 클라이언트에서 권한에 따라 제공한다. 로컬 제어, 파일 감시, OS 연동은 클라이언트의 네이티브 기능으로 처리한다.

고객 생산공장 운영에서는 일반 브라우저 직접 접근을 기본 사용 방식으로 두지 않는다. 실제 현장 사용자는 승인된 Windows WPF 클라이언트 앱을 통해 접근한다.

현장 입력은 처음부터 많은 직접 입력을 요구하지 않는다. 신호등식 기록, 기본 정형 문구, 짧은 메모, 관리자 대리 입력으로 시작하고, 현장 업무 방식이 바뀐 뒤 MES 연동과 자동화 입력으로 확장한다.

현장 의견 중 사진 기록, 짧은 코멘트, 작업순서판, 인수인계, 문서 보기 개선은 우선 반영한다. 메신저 전체 기능, 개인 메신저 수집, GPS 추적, 근태 관리, 전체 MES/ERP 대체, 개인 휴대폰 APK 또는 iOS 기본 배포는 현재 범위에서 제외한다.

현재 실증은 기존 현장 클라이언트와 통합하지 않고 별도 FlowNote Windows 클라이언트와 Python FastAPI 서버로 진행한다. 기존 MES 작업지시 또는 생산순서 정보는 필요 시 외부 참조 ID와 현장별 어댑터로 연결하되, 원본 시스템을 대체하지 않는다.

현장 관찰은 요구사항 분석과 실증의 일부이다. 작업자의 감각 판단, 보류, 재작업, 선별 이유처럼 MES 결과에 남지 않는 맥락을 기록 가능한 순간을 찾아야 하며, 단말기 위치와 작업자 동선도 함께 검증한다.

현장 코멘트는 원천 이력으로 시작하지만, 장기 활용을 위해서는 관리자급 분석과 보고서 문서화가 필요하다. 따라서 초기에는 원천 데이터와 추적성을 확보하고, 후속 단계에서 관리자 분석, AI 보조 보고서 초안, 최종 보고서 문서 저장을 확장한다.

배포는 사내 서버형을 기준으로 한다. FlowNote는 서버 PC 1대에 FastAPI 서버, SQLite DB, 로컬 storage 폴더를 두고 클라이언트 설치파일을 배포하는 방향으로 설계한다.

## 2. 단계별 구현 순서

### 1차: FastAPI + 네이티브 클라이언트 MVP

목표는 Python FastAPI 서버와 Windows WPF 클라이언트로 문서 관리의 핵심 흐름과 기본 현장지식 수집 흐름을 동작시키는 것이다.

- Python FastAPI 서버
- SQLite 기반 메타데이터 저장
- 서버 로컬 storage 파일 저장
- 회원 로그인
- 역할 기반 권한
- 관리자/현장 사용자 구분
- 문서 등록
- 고객 정의 문서 구조 생성
- 관리자 입력 기반 작업지시서 구조 생성
- 문서 구조 항목에 문서 연결
- 문서 버전 업로드
- 변경 사유 기록
- 문서 상태 관리
- 문서와 현장 코멘트 태그 관리
- 파일 저장소 저장
- 현장 공개 문서 열람
- 탐색기형 기본 검색
- 기본 현장 코멘트 등록
- 사진 첨부 기반 현장 기록
- 작업일지 또는 종이 기록 사진 등록
- 공정 중간 판단, 보류, 재작업, 선별 이유의 짧은 기록
- 코멘트 입력자와 실제 전달자 또는 작업자/작업그룹 기록
- 신호등식 상태 기록
- 기본 정형 문구 선택
- 관리자 대리 입력
- 문서 또는 문서 구조 항목과 코멘트 연결
- 관리자 입력 기반 작업순서판
- 현장 TV용 작업순서 화면
- 작업순서 변경 이력과 알림 이벤트
- 외부 시스템 참조 ID 저장
- 기본 연동 매핑 구조
- 배포 형태와 외부 접근 정책 설정
- API: `/api/v1`

목표는 현장 사용자 단말기 경험을 제공하는 것이다.

- Windows WPF 앱 UI
- 로그인 세션 연동
- 현장 공개 문서 열람
- 문서 변경 알림 확인
- 사진 첨부 기반 현장 기록
- 작업순서 확인
- 현장 코멘트 등록
- 정형 문구 선택
- 신호등식 기록
- 뷰어 자동 닫힘 적용
- 파일 다운로드 차단 유지

### 3차: 관리자 기능과 파일 감시 고도화

목표는 관리자 단말기와 파일 감시 업로드 보조를 제공하는 것이다.

- Windows WPF 관리자 UI
- 관리자 로그인
- 지정 파일 또는 폴더 감시
- 변경 후보 생성
- 새 버전 업로드 보조
- 변경 사유 입력 후 업로드 확정
- 관리자 다운로드와 로컬 파일 제어 정책 적용

- 관리자 코멘트 검토
- 원문/정리 문구 분리
- 관리자 분석 내용 작성
- 원천 코멘트와 작업내역을 보고서 근거로 연결
- 정형 문구 템플릿 관리
- 문서별 문제점 조회
- AI 보조 보고서 초안 생성
- 관리자 검토 후 보고서 확정
- 보고서 결과를 문서로 저장

### 4.5차: MES/ERP 연동 고도화

- 현장별 MES/ERP API 어댑터 구현
- 작업지시, 품목, 공정, 설비, 생산실적 수신
- 관리자 입력 작업지시와 외부 수신 작업지시 매핑
- 외부 작업지시와 작업순서판 항목 매핑
- 외부 원본 ID와 FlowNote 문서 구조/작업내역 매핑
- 기존 시스템이 없는 현장의 최소 수집 화면 구성
- 연동 실패와 동기화 이력 관리
- 실증 현장의 기존 MES 작업지시 또는 생산순서 정보와 작업순서판 연결 검토

### 5차: 검색과 AI

- 본문 추출 기반 검색 인덱스 구성
- 문서/버전/코멘트/보고서/작업내역/외부 정형 데이터 검색
- 작업지시 문서 위험 요소 미리보기
- 작업 조언 생성
- AI 응답 근거 표시
- 보고서 초안 작성 보조 고도화

### 장기 검토: 생산 데이터 기반 의사결정 조언

- 축적된 생산보고서와 작업내역 기반 패턴 분석
- 과거 유사 작업 비교
- 신규 작업 설계 참고 자료 추천
- 의사결정 조언
- 기존 문서 대장, 작업지시서, 관리 파일 파싱 지원

이 단계는 충분한 현장 데이터가 쌓인 뒤 검토하며, 현재 MVP나 1차 AI 범위로 보지 않는다.

## 3. 착수 원칙

- MVP는 기능을 빠르게 동작시키고 실제 사용 여부를 검증하는 것을 우선한다. 완성도 높은 최종 구조를 처음부터 확정하기보다, 현장에서 문서 열람, 기록, 검색, 작업순서 확인을 실제로 쓰는지 먼저 확인한다.
- 기술 검증만 별도로 오래 진행하기보다, 선택한 기술을 먼저 적용해 동작하는 기능을 만들고 구현 과정에서 한계와 보완점을 확인한다.
- 현장의 이야기는 가능한 한 제품 흐름에 반영한다. 다만 요청이 현재 개발 언어, 클라이언트 방향, 배포 방식, 보안 정책과 맞지 않으면 즉시 반영하지 않고 검토 후 결정한다.
- 기존 시스템과의 연동성은 초기부터 참조 ID와 매핑 구조를 준비하되, 현장별 상세 연동은 후속 단계에서 맞춘다.
- 장갑 낀 손, 터치 장비, 별도 현장 단말기 같은 사용 환경 판단은 초기 요구 수집 단계에서 과도하게 단정하지 않는다. 우선 현장의 표현과 요청을 기능으로 반영하고, 실제 사용 중 제약이 확인되면 UI와 입력 방식을 조정한다.
- 별도 장비 투입은 기능 검증을 위한 실험 범위로 본다. 공장이 비용을 투입하지 않는 전제에서는 장비 구매나 상시 운영 여부를 제품 기본값으로 두지 않고, 공장과 별도 회의 후 결정한다.
- 개인 휴대폰 설치는 현장 거부감과 설치 유도 비용이 크므로 현재 기본 배포 경로로 두지 않는다.
- UI는 가장 직관적인 흐름을 우선한다. 많은 설정과 고급 기능보다 사용자가 바로 열람, 검색, 기록, 확인할 수 있는 화면을 먼저 만들고 실제 사용 여부를 검증한다.
- 현장 프론트는 백엔드 복잡도와 별개로 단순하게 유지하고, 클릭 수와 입력 단계를 줄인다.
- 1차에서는 FastAPI 계약과 Windows WPF 앱의 핵심 화면 흐름을 함께 안정화한다.
- 다운로드 차단과 뷰어 자동 닫힘은 클라이언트 앱과 서버 감사 로그를 함께 사용해 구현한다.
- 1차에서는 최소 현장 코멘트 수집과 작업자/작업그룹 추적을 포함하되, 관리자 분석/정형 문구 템플릿 고도화/보고서는 후속 단계로 분리한다.
- 1차에서는 기본 검색과 작업순서판을 포함하되, 본문 추출 기반 고급 검색과 AI 검색은 후속 단계로 분리한다.
- CAD 원본 직접 뷰어와 CAD-PDF 자동 변환은 후속 단계로 두고, 초기에는 PDF 변환본 또는 지원 가능한 미리보기 계층을 우선한다.
- Windows WPF 클라이언트는 앱 UI를 기준으로 구현한다.
- 로컬 기능은 앱에서 필요한 범위가 명확해진 뒤 네이티브 기능으로 구현한다.
- AI와 보고서는 문서/권한/열람 보안이 안정화된 뒤 구현한다.
- 현장 입력은 업무 흐름을 방해하지 않는 수준에서 시작하고, 관리자 주도 운영과 교육으로 단계적으로 정착시킨다.
- 현장 방문, 설치, 교육은 생산 흐름을 방해하지 않는 시간과 방식으로 진행하며, 작업자 프라이버시와 감시 오해를 피한다.
- 현장 관찰 결과가 보고서 한 줄로 끝나더라도 이후 기능 판단의 근거로 남긴다.
